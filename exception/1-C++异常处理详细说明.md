# C++异常处理详细说明

异常处理是现代 C++ 中不可或缺的一部分，它提供了一种结构化的方式来处理程序运行时发生的错误和异常情况。本文将全面介绍 C++ 的异常处理机制，包括基本语法、高级特性、最佳实践以及常见陷阱

## 1. 异常处理概述

### 1.1 为什么需要异常处理

在软件开发中，错误和异常情况不可避免。传统错误处理方式（如返回错误码）存在明显缺陷：

- **错误传播繁琐**：每个函数都必须检查返回值并向上传递，导致代码膨胀；

- **错误码容易被忽略**：调用者可能忘记检查错误码，导致程序继续执行在错误状态；

- **正常逻辑与错误处理混杂**：代码可读性差，难以维护；

- **资源管理困难**：在多层调用中，一旦某层出错，需要确保已分配的资源正确释放

C++异常处理机制提供了一种结构化的方式，将错误检测与错误处理分离，允许程序在发生异常时自动清理栈上的资源，并将控制权转移到能够处理该错误的上下文中。


### 1.2 异常的基本思想

- **抛出(throw)**: 当检测到错误时，程序抛出一个异常对象

- **传播(propagation)**: 异常沿着调用链向上传播，直到找到匹配的处理器

- **捕获(catch)**: 异常处理器（catch块）捕获并处理异常

- **栈展开(stack unwinding)**: 在异常传播过程中，所有局部对象会被自动销毁（析构函数调用），保证资源释放。


## 2. 异常处理的基本语法

### 2.1 try 块和 catch 块

```c++
try {
    // 可能抛出异常的代码
} catch (类型1 参数1) {
    // 处理类型1的异常
} catch (类型2 参数2) {
    // 处理类型2的异常
} catch (...) {
    // 捕获所有其他类型的异常（catch-all）
}
```

- `try`关键字开始一个可能抛出异常的代码块

- 一个`try`块后面必须紧跟一个或多个 catch 块;

- `catch`块按照出现的顺序进行匹配，第一个匹配的块被执行;

- `catch (...)` 捕获任何类型的异常，通常用于资源清理后重新抛出;

### 2.2 throw表达式

```c++
throw 表达式;
```

`throw`会创建一个异常对象，其类型由表达式的静态类型决定。异常对象是原始表达式的拷贝（可能会经过调整，例如数组退化为指针）。

**示例**:

```c++
throw 42;                     // 抛出 int 类型的异常
throw "error";                 // 抛出 const char* 类型的异常
throw std::runtime_error("..."); // 抛出 std::runtime_error 类型的异常
```

抛出后，程序立即开始寻找匹配的`catch`，后续代码不再执行。

### 2.3 匹配规则

catch 块匹配异常对象的类型，规则如下：

1. 允许从派生类到基类的转换（按引用或按值）

1. 允许从非 const 到 const 的转换（如果 catch 是 const 引用）

1. 允许从派生类指针到基类指针的转换（对于指针类型的异常）

1. 不允许标准算术转换或用户定义转换


**注意**: 如果按值捕获，会发生对象切片，丢失派生类信息。因此推荐使用引用捕获

```c++
catch (std::exception& e)      // OK，引用，避免切片
catch (std::exception e)       // 不推荐，对象切片
```

### 2.4 重新抛出

在`catch`块中，可以使用不带参数的 throw 重新抛出当前捕获的异常，保留原始异常类型：

```c++
try {
    // ...
} catch (const std::exception& e) {
    // 记录日志
    throw;  // 重新抛出
}
```
如果只想部分处理异常，然后交由上层处理，可以使用重新抛出。


## 3. 异常类型

### 3.1 标准异常类

C++ 标准库定义了一组异常类，全部派生自`std::exception`，定义在 `<exception>`、`<stdexcept>`、`<new>`、`<typeinfo>`等头文件中。

```txt
std::exception
    ├── std::bad_alloc           (new 失败)
    ├── std::bad_cast            (dynamic_cast 失败)
    ├── std::bad_typeid          (typeid 作用于空指针)
    ├── std::bad_exception       (意外异常)
    ├── std::logic_error         (程序逻辑错误，可避免)
    │   ├── std::invalid_argument
    │   ├── std::domain_error
    │   ├── std::length_error
    │   └── std::out_of_range
    └── std::runtime_error       (运行时环境错误，不可预测)
        ├── std::range_error
        ├── std::overflow_error
        └── std::underflow_error
```

所有标准异常都提供 what() 成员函数，返回一个描述错误的 C 风格字符串。

### 3.2 自定义异常类

自定义异常类通常从 std::exception 或其派生类继承，并覆盖 what() 函数。

```c++
#include <exception>
#include <string>

class MyException : public std::exception {
public:
    explicit MyException(const char* msg) : message_(msg) {}
    const char* what() const noexcept override { return message_.c_str(); }
private:
    std::string message_;
};
```

更简单的做法是从`std::runtime_error`派生，它已经处理了字符串存储和 what() 实现：

```c++
#include <stdexcept>

class MyException : public std::runtime_error {
public:
    explicit MyException(const std::string& msg) : std::runtime_error(msg) {}
};
```

这样，自定义异常类可以直接用于`throw`，并且能被`catch (const std::exception&)`捕获。


## 4. 栈展开与 RAII

### 4.1 栈展开的过程

当异常被抛出后，程序开始在当前`try`块中查找匹配的`catch`。如果找不到，则退出当前函数，并销毁所有局部对象(自动存储器对象)，然后继续在调用函数中查找。这个过程称为`栈展开`。

栈展开保证了在退出每个作用域时，所有局部对象的析构函数都会被调用。这是 C++ 异常处理机制与资源管理紧密耦合的关键点。

### 4.2 RAII与资源管理

RAII（Resource Acquisition Is Initialization）是 C++ 管理资源的惯用法：资源（如内存、文件句柄、互斥锁）在构造函数中获取，在析构函数中释放。由于栈展开会调用析构函数，RAII 对象可以确保即使在异常发生时，资源也能被正确释放。

**示例**: 自动关闭文件

```c++
#include <fstream>
#include <stdexcept>

void processFile(const std::string& name) {
    std::ifstream file(name);   // RAII: 构造时打开文件
    if (!file)
        throw std::runtime_error("cannot open file");
    // 处理文件，可能抛出异常
    // ...
    // 函数退出时，file 析构，自动关闭文件
}
```

无论是否抛出异常，`std::ifstream`的析构函数都会被执行，文件被正确关闭。

**重要**: 如果手动管理资源（如使用 new/delete），在异常路径中容易泄漏。始终优先使用 RAII 封装资源（智能指针、标准容器等）。

## 5. 异常安全性

异常安全性衡量代码在抛出异常时维持正确行为的能力。通常分为三个级别：

### 5.1 基本保证（Basic Guarantee）

- 抛出异常后，程序处于有效但未指定的状态，没有资源泄漏。

- 所有对象的不变量（invariants）仍然保持，但数据可能已被修改。

例如，一个容器在插入元素时抛出异常，容器可能处于不一致的状态，但仍然是有效的（可以安全销毁，不会泄漏内存）。

### 5.2 强保证（Strong Guarantee）

- 操作要么完全成功，要么完全失败（无副作用）。

- 如果抛出异常，程序状态与调用操作前完全相同（原子性）。

通常通过“拷贝-交换”惯用法实现：先在副本上执行可能失败的操作，如果成功，则用副本替换原对象


**示例**：拷贝-交换实现强保证

```c++
class Widget {
public:
    void update(const Data& newData) {
        Data temp = newData;       // 拷贝
        temp.validate();           // 可能抛出异常
        swap(data_, temp);         // 无异常交换
    }
private:
    Data data_;
};
```

如果 validate() 抛出异常，temp 被销毁，原对象`data_`保持不变。

### 5.3 不抛异常保证（Nothrow Guarantee）

- 承诺操作绝不会抛出异常。

- 适用于析构函数、移动操作、swap 函数、某些基本类型的操作等。

- C++11 引入 noexcept 关键字来声明这种保证。

### 5.4 如何编写异常安全代码

- 使用 RAII 管理所有资源。

- 在可能抛出异常的操作前，尽量在副本上操作，然后无异常地交换。

- 将可能抛出异常的代码与修改状态的代码分离。

- 保证析构函数、swap、移动操作不抛出异常（通常标记为 noexcept）


## 6. noexcept说明符

###  6.1 基本语法
C++11 引入 `noexcept` 关键字，用于指定函数是否会抛出异常。

```c++
void f() noexcept;           // 承诺不抛出异常
void g() noexcept(false);    // 可能抛出异常（默认）
```

如果 noexcept 函数抛出了异常（即异常逃逸出函数），程序会调用 std::terminate 终止。

### 6.2 为什么使用 noexcept

- 优化：编译器可以生成更高效的代码，因为无需维护栈展开所需的额外信息（例如，可能不需要生成异常表）。

- 标准库优化：标准库容器在重新分配时，会优先使用 noexcept 的移动构造函数以保证强异常安全。如果没有 noexcept，可能会退而使用拷贝构造。

- 表达意图：明确告知调用者和维护者，该函数不会抛出异常

### 6.3 noexcept 运算符

noexcept 还可以作为运算符，在编译时检查表达式是否会抛出异常：

```c++
void h();
std::cout << noexcept(h()) << std::endl;  // 输出 0 (false)

void i() noexcept;
std::cout << noexcept(i()) << std::endl;  // 输出 1 (true)
```

常用于模板元编程和条件 noexcept 声明。

### 6.4 条件 noexcept

可以使用布尔常量表达式条件性地指定 noexcept:

```c++
template<typename T>
void swap(T& a, T& b) noexcept(noexcept(T::swap(a, b))) {
    a.swap(b);
}
```

这里，swap 的 noexcept 状态取决于 T::swap 是否 noexcept

### 6.5 析构函数与 noexcept

C++11 开始，析构函数默认是 noexcept 的（即使未显式标记）。如果析构函数抛出异常且未被捕获，程序会终止。因此，析构函数中绝不应抛出向外传播的异常。

## 7. 构造函数与析构函数中的异常

### 7.1 构造函数抛出异常

- 如果构造函数抛出异常，对象的析构函数不会被调用（因为对象尚未完全构造）。

- 但是，已经构造完成的成员对象和基类子对象会被正确析构（它们的析构函数会被调用）。

因此，构造函数中如果使用原始指针分配资源，必须在抛出前手动释放。更好的做法是使用 RAII 成员（如智能指针、容器），让它们自动管理资源。

```c++
class Bad {
public:
    Bad() {
        p = new int(10);
        throw std::runtime_error("error"); // 泄漏 p
    }
private:
    int* p;
};

class Good {
public:
    Good() : p(std::make_unique<int>(10)) {
        throw std::runtime_error("error"); // OK, unique_ptr 会自动释放
    }
private:
    std::unique_ptr<int> p;
};
```

### 7.2 析构函数抛出异常
析构函数抛出异常是非常危险的：

- 如果在栈展开期间析构函数抛出异常，而异常正在传播中，会导致 std::terminate 被调用（两个异常同时存在）。

- 如果在非异常情况下析构函数抛出异常，且未被捕获，同样会导致 std::terminate。

因此，析构函数必须捕获所有异常并吞没或处理，绝不能向外传播。如果需要在析构函数中执行可能抛出异常的操作，应使用 try-catch 块包围。

```c++
class Resource {
public:
    ~Resource() {
        try {
            cleanup(); // 可能抛出异常
        } catch (...) {
            // 记录日志，但不要重新抛出
        }
    }
};
```

C++11 起，析构函数默认隐式`noexcept(true)`，如果显式指定`noexcept(false)`且抛出了异常，程序仍可能终止（取决于上下文）。最佳实践：让析构函数永不抛出。

## 8. 异常规范的历史与现状

### 8.1 动态异常规范（C++98/03）
C++98 引入了动态异常规范，用于声明函数可能抛出的异常类型：

```c++
void f() throw(std::runtime_error, std::bad_alloc); // 可能抛出这两种异常
void g() throw();                                   // 不抛出任何异常
```

但这种机制在运行时检查，效率低下，且在实践中很少使用。C++11 将其弃用，C++17 中移除了动态异常规范（但仍保留了 throw() 作为 noexcept 的同义词，直到 C++20 移除）。

### 8.2 noexcept 代替 throw()

C++11 引入 noexcept，提供编译时检查，取代了动态异常规范。throw() 在 C++17 中被视为 noexcept 的同义词，但已在 C++20 中移除。现在应只使用 noexcept。

## 9. 性能考虑
异常处理通常被认为有性能开销，但现代 C++ 实现采用了零成本异常模型（zero-cost exception handling）：在没有异常抛出时，正常执行路径没有额外开销；只有在抛出异常时，才会产生较大的成本（查找处理代码、栈展开等）。

- 正常路径：无开销（可能少数情况会有少许静态开销）。

- 异常路径：开销较大（但异常本就用于处理错误，不应频繁发生）。

因此，不应使用异常来控制普通流程（如循环终止），但处理真正的错误时，性能影响可接受。

注意：某些嵌入式环境可能禁用异常以节省代码空间；此时可使用编译选项`-fno-exceptions`（GCC/Clang）或 `/EHs-c-`（MSVC）。


## 10. 跨函数与跨线程异常传播

### 10.1 异常跨函数传播
异常会沿着调用链向上传播，直到被捕获。如果直到 main 都没有捕获，程序调用`std::terminate`终止。

### 10.2 线程间的异常传播
C++11 提供了`std::exception_ptr`，允许在不同线程之间传递异常。

- `std::current_exception()`：在 catch 块中调用，返回一个 exception_ptr，指向当前处理的异常（或空，如果没有异常）。

- `std::rethrow_exception(eptr)`：重新抛出 exception_ptr 所指向的异常。

- `std::make_exception_ptr(e)`：直接创建一个 exception_ptr，指向异常对象 e 的拷贝。

典型用法：在工作线程中捕获异常，保存到 exception_ptr，然后主线程检查并重新抛出。

```c++
#include <iostream>
#include <thread>
#include <exception>

void worker(std::exception_ptr& eptr) {
    try {
        throw std::runtime_error("worker error");
    } catch (...) {
        eptr = std::current_exception(); // 保存异常
    }
}

int main() {
    std::exception_ptr eptr;
    std::thread t(worker, std::ref(eptr));
    t.join();

    if (eptr) {
        try {
            std::rethrow_exception(eptr);
        } catch (const std::exception& e) {
            std::cerr << "Caught: " << e.what() << '\n';
        }
    }
    return 0;
}
```

### 10.3 std::nested_exception

C++11 提供了`std::nested_exception`，允许在一个异常中嵌套另一个异常，便于保留错误上下文。通过`std::throw_with_nested`抛出嵌套异常，然后使用`dynamic_cast`或`std::rethrow_if_nested`访问嵌套异常。

## 11. 常见陷阱与最佳实践

### 11.1 不要按值捕获异常
按值捕获会导致对象切片，丢失派生类信息。应使用引用捕获。

```c++
catch (std::exception e)          // 错误：切片
catch (const std::exception& e)   // 正确
```

### 11.2 不要在析构函数中抛出异常

前面已详细说明。如果必须，捕获并吞没。

### 11.3 避免在 noexcept 函数中抛出异常
如果 noexcept 函数抛出异常，程序会直接终止，可能无法进行任何清理。确保 noexcept 函数内部不会抛出异常。

### 11.4 不要在异常对象中动态分配内存

异常对象本身是拷贝构造的，如果异常对象内部包含动态分配的内存，可能抛出 std::bad_alloc，导致原异常丢失。建议异常对象只包含简单信息或使用 std::string（它可能分配内存，但标准库实现通常会保证不会失败，或者失败时会抛出另一个异常，造成混乱）。最好使用静态字符串或从 std::exception 派生。

### 11.5  使用 RAII 管理所有资源
这是 C++ 异常安全的核心。所有动态资源（内存、锁、文件等）都应包装在 RAII 对象中。

### 11.6 不要将异常用于控制流
异常是为错误处理设计的，开销较大。正常的流程控制（如循环结束）应使用条件语句.

### 11.7 保持异常类简单
异常对象会在栈展开时被拷贝，可能多次拷贝（取决于实现）。尽量使异常类轻量，避免大量数据或复杂构造。

### 11.8 在 catch 块中按需处理，不要吞没异常
除非确实需要，否则不要无声地忽略异常。至少应记录日志，或重新抛出。

### 11.9 使用 noexcept 标记移动操作和 swap
标准库容器依赖移动操作的 noexcept 来提供强异常安全保证。标记为 noexcept 能让你的类型更好地与标准库协作。

### 11.10 注意函数 try 块

函数 try 块允许捕获构造函数初始化列表中的异常，但需要注意，捕获后重新抛出或终止是唯一选择，因为对象尚未完全构造。

```c++
C::C() try : member() {
    // 构造函数体
} catch (...) {
    // 初始化列表中的异常可以在此捕获，但必须重新抛出或终止
    throw;
}
```

## 12. 总结
C++ 异常处理机制是一种强大而优雅的错误处理方式，与 RAII 结合可以实现自动资源管理和异常安全。理解异常的原理、语法和最佳实践，能够编写出更健壮、更易维护的代码。

- 基本结构：try/throw/catch。

- 异常类型：使用标准异常或自定义派生类。

- 栈展开：自动销毁局部对象，RAII 是基础。

- 异常安全：基本、强、不抛异常三个级别。

- noexcept：声明不抛异常的函数，利于优化。

- 线程间异常传递：使用 std::exception_ptr。

- 最佳实践：引用捕获、析构函数不抛、RAII、避免控制流异常等。
